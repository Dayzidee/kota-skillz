# Idempotent Sink (Tier 1)

## When to Use
- **Exactly-Once Delivery**: When you must ensure that a message from a queue (e.g., Kafka) is processed exactly once by a database, even if the worker crashes and restarts.
- **Webhook Handlers**: When an external service (e.g., Stripe) might send the same event multiple times.

## The Pattern
Design the "Sink" (the database or API) to ignore duplicate requests by using a unique identifier from the source.

### Implementation Logic
1. **The Unique ID**: Every incoming message must have a unique, persistent ID (e.g., Kafka Offset, Stripe Event ID, or a UUID generated by the producer).
2. **The Atomic Update**: When updating the database, include the ID in the same transaction.
   - Example (SQL): `UPDATE balances SET amount = amount + 10, last_offset = 456 WHERE user_id = 1 AND last_offset < 456;`
3. **The Check**: If the `last_offset` in the database is already greater than or equal to the incoming offset, the update is ignored.

## Trade-offs
- **Storage**: You must store the `last_processed_id` for every entity or partition.
- **Deterministic**: The source must provide the same ID for the same logical operation every time it retries.

## Failure Modes
- **Non-Deterministic IDs**: If the producer generates a new UUID for every retry, the consumer cannot detect duplicates.
- **Race Conditions**: If multiple workers try to process the same message without a "Compare-and-Set" or "Fencing" mechanism, you might get double-processing.

## Verification
- **Replay Test**: Manually send the same message 10 times. Verify that the database state only changes once.
- **Offset Audit**: Verify that the `last_processed_offset` in the database matches the consumer's committed offset in Kafka.
